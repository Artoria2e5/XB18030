Abstract

WTF-8 (Wobbly Transformation Format − 8-bit) is a superset of UTF-8 that encodes surrogate code points if they are not in a pair. It represents, in a way compatible with UTF-8, text from systems such as JavaScript and Windows that use UTF-16 internally but don’t enforce the well-formedness invariant that surrogates must be paired.

1. Background and motivation

This section is non-normative.

When Unicode 1.0 was published in 1991, it defined 65536 code points from U+0000 to U+FFFF and assigned characters to around half of them. Many software implementations chose the obvious memory reprentation for Unicode text of 16 bits per code point / character.

As subsequent versions of Unicode assigned more characters, it became apparent that 65536 code points would not be sufficient. Unicode was extended to 1114111 code points from U+0000 to U+10FFFF, and the UTF-16 encoding was introduced. This encoding preserves compatibility with existing 16-bit based systems and represents new (supplementary) code points as a pair of “surrogates”.

UTF-16 is designed to represent any Unicode text, but it can not represent a surrogate code point pair since the corresponding surrogate code unit pairs would instead represent a supplementary code point. Therefore, the concept of Unicode scalar value was introduced and Unicode text was restricted to not contain any surrogate code point. (This was presumably deemed simpler that only restricting pairs.)

UTF-16 was redefined to be ill-formed if it contains unpaired surrogate code units. UTF-8 was similarly redefined to be ill-formed if it contains surrogate byte sequences.

Meanwhile, 16-bit based systems had little to no incentive to do anything about surrogates: For several years, Unicode did not assign any character to supplementary code points, and then (until emoji) only comparatively rare characters. Additionally, the Unicode Standard does not require conforming implementations to maintain well-formedness of UTF-16 strings.

As a result, surrogates do occur in practice and need to be preserved. For example:

  • In ECMAScript (a.k.a. JavaScript), a String value is defined as a sequence of 16-bit integers that usually represents UTF-16 text but may or may not be well-formed.
  • Windows applications normally use UTF-16, but the file system treats path and file names as an opaque sequence of WCHARs (16-bit code units).

This specification calls this practice WTF-16.

Unpaired surrogate code units are the only case where an arbitrary sequence of code units is ill-formed in UTF-16. UTF-8, however, is more complex and maintaining its well-formedness is arguably more valuable.

This specification defines WTF-8, a superset of UTF-8 that can losslessly represent arbitrary code unit sequences (even if ill-formed in UTF-16) but preserves the other well-formedness constraints of UTF-8.

2. Terminology

These definitions correspond to those of the Glossary of Unicode Terms [UNICODE], except for code unit which is slightly different for the purpose of this specification.

A Unicode code point is any value in the Unicode codespace; that is, the range of integers from 0 to 1114111. It is noted with a “U+” prefix and four to six hexadecimal digits: the first and last code points are U+0000 and U+10FFFF.

The Basic Multilingual Plane is the range of code points from U+0000 to U+FFFF.

A BMP code point is a code point in the Basic Multilingual Plane.

A supplementary code point is a code point not in the Basic Multilingual Plane. That is, a code point in the range from U+10000 to U+10FFFF.

A Unicode scalar value is a code point that is not a surrogate code point. That is, a code point in the range from U+0000 to U+D7FF, or in the range U+E000 to U+10FFFF.

A BMP scalar value is a Unicode scalar value in the Basic Multilingual Plane. That is, a code point in the range from U+0000 to U+D7FF, or in the range U+E000 to U+FFFF.

Unicode text is a sequence of Unicode scalar values.

UTF-8 is an encoding of Unicode text using 8-bit bytes. Each Unicode scalar value is represented as a sequence of one to four bytes.

UTF-16 is an encoding of Unicode text using 16-bit code units. BMP scalar values are represented as a single code unit with the same value. Supplementary code points are represented as a surrogate code unit pair.

Note: this specification is only concerned with the UTF-16 encoding form (based on code units), and not with the encoding scheme (based on bytes, with UTF-16BE and UTF-16LE variants).

A string is well-formed (not ill-formed) in a given encoding if it follows the specification of that encoding. [UNICODE] defines Well-Formed Code Unit Sequence for UTF-8 and UTF-16.

In particular:
  • Unpaired surrogate code units are ill-formed in UTF-16.
  • Surrogate byte sequences are ill-formed in UTF-8.

The replacement character is the code point U+FFFD REPLACEMENT CHARACTER (�). It is used as a substitute to replace ill-formed sub-sequences during a conversion.

A code unit, for the purpose of this specification, is a 16-bit integer used in UTF-16. It is noted with a “0x” prefix and four hexadecimal digits: the first and last code units are 0x0000 and 0xFFFF.

Note: The byte serialization or memory representation of a code unit (little-endian or big-endian) is out of scope for this specification.

Note: Unicode also uses this term for 8-bit and 32-bit code units in the UTF-8 and UTF-32 encodings.

When an algorithm iterates over a sequence (“For every i in …”), consuming the next item means advancing in the sequence such that that item will be skipped during the following iteration of the loop: the item after the next becomes the next item.

The following algorithm prints “1”, “2”, and “4”.

For every digit i in “1234”, run these substeps:

  1. Print i
  2. If i is 2, consume the next digit.

2.1. Surrogate code points

A lead surrogate code point or high surrogate code point is a code point in the range from U+D800 to U+DBFF.

A trail surrogate code point or low surrogate code point is a code point in the range from U+DC00 to U+DFFF.

A surrogate code point is either a lead surrogate code point or a trail surrogate code point. That is, a code point in the range from U+D800 to U+DFFF.

A surrogate code point pair is a sequence of a lead surrogate code point followed by a trail surrogate code point.

An unpaired surrogate code point is a surrogate code point that is not part of a surrogate code point pair.

2.2. Surrogate code units

A lead surrogate code unit or high surrogate code unit is a code unit in the range from 0xD800 to 0xDBFF.

A trail surrogate code unit or low surrogate code unit is a code unit in the range from 0xDC00 to 0xDFFF.

A surrogate code unit is either a lead surrogate code unit or a trail surrogate code unit. That is, a code unit in the range from 0xD800 to 0xDFFF.

A surrogate code unit pair is a sequence of a lead surrogate code unit followed by a trail surrogate code unit. In UTF-16, it represents a supplementary code point.

An unpaired surrogate code unit is a surrogate code unit that is not part of a surrogate code unit pair.

2.3. Surrogate byte sequences

A lead surrogate byte sequence or high surrogate byte sequence is a sequence of three bytes that represents a lead surrogate code point in generalized UTF-8.

A trail surrogate byte sequence or low surrogate byte sequence is a sequence of three bytes that represents a trail surrogate code point in generalized UTF-8.

A surrogate byte sequence is either a lead surrogate byte sequence or a trail surrogate byte sequence. That is, a sequence of three bytes that represents a surrogate code point in generalized UTF-8.

Table 1. Surrogate byte sequences

Bytes noted in hexadecimal.

First byte Second byte Third byte
Lead surrogate byte sequence ED A0 to AF 80 to BF
Trail surrogate byte sequence ED B0 to BF 80 to BF
Surrogate byte sequence ED A0 to BF 80 to BF

Note: A surrogate byte sequence is ill-formed in UTF-8. Decoders are required to treat is as an error

A surrogate pair byte sequence is a sequence six bytes composed of a lead surrogate byte sequence followed by a trail surrogate byte sequence.

An unpaired surrogate byte sequence is a surrogate byte sequence that is not part of a surrogate pair byte sequence.

3. Generalized UTF-8

For the purpose of this specification, generalized UTF-8 is an encoding of sequences of code points (not restricted to Unicode scalar values) using 8-bit bytes, based on the same underlying algorithm as UTF-8. It is a strict superset of UTF-8 (like UTF-8 is a strict superset of ASCII).

Each code point is encoded as a sequence of one to four bytes:

Table 2. Bit distribution

Bytes noted in binary, most significant bit first. x bits represent the least significant bits of the code points.

Code point First byte Second byte Third byte Fourth byte
U+0000 to U+007F 0xxxxxxx
U+0080 to U+07FF 110xxxxx 10xxxxxx
U+0800 to U+FFFF 1110xxxx 10xxxxxx 10xxxxxx
U+10000 to U+10FFFF 1110xxxx 10xxxxxx 10xxxxxx 10xxxxxx

A byte sequence is well-formed in generalized UTF-8 if and only if:

  • It is the empty string, or
  • It is composed of a byte sequence that is well-formed in generalized UTF-8 followed by a byte sequence in the following table.

Table 3. Well-formed byte sequences representing a single code point

Bytes noted in hexadecimal.

Code point First byte Second byte Third byte Fourth byte
U+0000 to U+007F 00 to 7F
U+0080 to U+07FF C2 to DF 80 to BF
U+0800 to U+0FFF E0 A0 to BF 80 to BF
U+1000 to U+FFFF E1 to EF 80 to BF 80 to BF
U+10000 to U+3FFFF F0 90 to BF 80 to BF 80 to BF
U+40000 to U+FFFFF F1 to F3 80 to BF 80 to BF 80 to BF
U+100000 to U+10FFFF F4 80 to 8F 80 to BF 80 to BF

4. The WTF-16 encoding

WTF-16 (Wobbly Transformation Format − 16-bit) is an encoding of sequences of code points that do not contain any surrogate code point pair using 16-bit code units.

Note: Like UTF-16, WTF-16 can not represent a surrogate code point pair since the corresponding surrogate code unit pair would instead represent a supplementary code point.

It is identical to UTF-16, with the well-formedness constraint removed and the added semantics that an unpaired surrogate code unit represents the corresponding surrogate code point. It is a strict superset of UTF-16.

WTF-16 has no well-formedness constraint. Any sequence of code units is well-formed in WTF-16.

4.1. Encoding to WTF-16

To encode from a sequence of code points that does not contain any surrogate code point pair to WTF-16, run these steps:

  1. Let result be a sequence of code units, initially empty.
  2. For every code point P of the input, run these substeps:
    1. If P is a supplementary code point, run these substeps:
      1. Append to result a code unit of value ((P - 0x10000) >> 10) + 0xD800
      2. Append to result a code unit of value ((P - 0x10000) & 0x3FF) + 0xDC00
    2. Otherwise (P is a BMP code point), append to result a code unit of value P.
  3. Return result.

Note: If the input is restricted to Unicode text, this is identical to encoding UTF-16 and the resulting sequence is well-formed in UTF-16.

If, on the other hand, the input does contain a surrogate code point pair, the conversion will be incorrect and the resulting WTF-16 string will not represent the original code points.

This situation should be considered an error, but this specification does not define how to handle it. Possibilities include aborting the conversion, or replacing one of the surrogate code points of the pair with a replacement character.

4.2. Decoding from WTF-16

To decode from WTF-16 to a sequence of code points that does not contain any surrogate code point pair, run these steps:

  1. Let result be a sequence of code points, initially empty.
  2. For every code unit U of the input, run these substeps:
    1. If U is a lead surrogate code unit and is not the last code unit of the input, run these substeps:
      1. Let next be the next code unit of the input, and consume it.
      2. If next is a trail surrogate code unit, append to result a code point of value 0x10000 + ((lead surrogate - 0xD800) << 10) + (U - 0xDC00).
      3. Otherwise, append to result a code point of value U, then a code point of value next.
    2. Otherwise, append to result a code point of value U.
  3. Return result.

Note: If the input is well-formed in UTF-16, this is identical to decoding UTF-16 and the resulting sequence is Unicode text.

5. The WTF-8 encoding

WTF-8 (Wobbly Transformation Format − 8-bit) is an encoding of sequences of code points that do not contain any surrogate code point pair using 8-bit bytes.

Note: Like UTF-8 is artificially restricted to Unicode text in order to match UTF-16, WTF-8 is artificially restricted to exclude surrogate code point pairs in order to match WTF-16.

It is identical to generalized UTF-8, with the additional well-formedness constraint that a surrogate pair byte sequence is ill-formed. It is a strict subset of generalized UTF-8 and a strict superset of UTF-8.

Note: Similarly, UTF-8 is a strict superset of ASCII.

5.1. Encoding to WTF-8

To encode from a sequence of code points that does not contain any surrogate code point pair to WTF-8, run these steps:

  1. Let result be a sequence of bytes, initially empty.
  2. For every code point P of the input, depending on P:
    U+0000 to U+007F
    Append to result one byte of value P.
    U+0080 to U+07FF
    Append to result two bytes of values:
    1. 0xC0 | (P >> 6)
    2. 0x80 | (P & 0x3F)
    U+0800 to U+FFFF
    Append to result three bytes of values
    1. 0xC0 | (P >> 12)
    2. 0x80 | ((P >> 6) & 0x3F)
    3. 0x80 | (P & 0x3F)
    U+FFFF to U+10FFFF
    Append to result four bytes of values
    1. 0xC0 | (P >> 18)
    2. 0x80 | ((P >> 12) & 0x3F)
    3. 0x80 | ((P >> 6) & 0x3F)
    4. 0x80 | (P & 0x3F)
  3. Return result.

If the input does contain a surrogate code point pair, the resulting sequence will be ill-formed in WTF-8.

This situation should be considered an error, but this specification does not define how to handle it. Possibilities include aborting the conversion, or replacing one of the surrogate code points of the pair with a replacement character.

5.2. Decoding from WTF-8

To decode from WTF-8 to a sequence of code points that does not contain any surrogate code point pair, run these steps:

  1. Let result be a sequence of code points, initially empty.
  2. For every byte B of the input, depending on B:
    0x00 to 0x7F
    Append to result a code point of value B.
    0xC2 to 0xDF

    If there is no input after B, append a replacement character to result and abort this iteration of the loop.

    Let B2 be the next byte and consume it.

    If B2 is not in the range from 0x80 to 0xBF, append a replacement character to result and abort this iteration of the loop.

    Append to result a code point of value ((B & 0x1F) << 6) + (B2 & 0x3F)

    0xE0 to 0xEF

    If there is no input after B, append a replacement character to result and abort this iteration of the loop.

    Let B2 be the next byte and consume it.

    If B2 is not in the range from 0x80 to 0xBF, append a replacement character to result and abort this iteration of the loop.

    If B is 0xE0 and B2 is not in the range from 0xA0 to 0xBF, append a replacement character to result and abort this iteration of the loop.

    If there is no input after B2, append a replacement character to result and abort this iteration of the loop.

    Let B3 be the next byte and consume it.

    If B3 is not in the range from 0x80 to 0xBF, append a replacement character to result and abort this iteration of the loop.

    Append to result a code point of value ((B & 0x0F) << 12) + ((B2 & 0x3F) << 6) + (B3 & 0x3F)

    0xF0 to 0xF4

    If there is no input after B, append a replacement character to result and abort this iteration of the loop.

    Let B2 be the next byte and consume it.

    If B2 is not in the range from 0x80 to 0xBF, append a replacement character to result and abort this iteration of the loop.

    If B is 0xF0 and B2 is not in the range from 0x90 to 0xBF, append a replacement character to result and abort this iteration of the loop.

    If B is 0xF4 and B2 is not in the range from 0x80 to 0x8F, append a replacement character to result and abort this iteration of the loop.

    If there is no input after B2, append a replacement character to result and abort this iteration of the loop.

    Let B3 be the next byte and consume it.

    If B3 is not in the range from 0x80 to 0xBF, append a replacement character to result and abort this iteration of the loop.

    If there is no input after B3, append a replacement character to result and abort this iteration of the loop.

    Let B4 be the next byte and consume it.

    If B4 is not in the range from 0x80 to 0xBF, append a replacement character to result and abort this iteration of the loop.

    Append to result a code point of value ((B & 0x07) << 18) + ((B2 & 0x3F) << 12) + ((B3 & 0x3F) << 6) + (B4 & 0x3F)

    Anything else
    Append a replacement character to result.
  3. Return result.

Note: If the input is well-formed in UTF-8, this is identical to decoding UTF-8 and the resulting sequence is Unicode text.

5.3. Converting between WTF-8 and WTF-16

To convert from WTF-16 to WTF-8, run these steps:

  • Decode from WTF-16 to a sequence of code points that does not contain any surrogate code point pair
  • Encode from a sequence of code points that does not contain any surrogate code point pair to WTF-8

Note: This conversion never fails and is lossless.

To convert from WTF-8 to WTF-16, run these steps:

  • Decode from WTF-8 to a sequence of code points that does not contain any surrogate code point pair
  • Encode from a sequence of code points that does not contain any surrogate code point pair to WTF-16

Note: This conversion never fails and, if the input is well-formed in WTF-8, is lossless.

5.4. Converting between WTF-8 and UTF-8

Since WTF-8 is a superset of UTF-8, any sequence of byte that is well-formed in UTF-8 is also well-formed in WTF-8 and represents the same text. To convert from UTF-8 to WTF-8, return the input unchanged.

Note: This conversion never fails and is losseless.

To convert lossily from WTF-8 to UTF-8, replace any surrogate byte sequence with <0xEF, 0xBF, 0xBD>, the sequence of three bytes that encode the replacement character in UTF-8.

Note: This conversion never fails but is lossy.

To convert strictly from WTF-8 to UTF-8, run these steps:

  1. If the input contains a surrogate byte sequence, return failure.
  2. Otherwise, return the input unchanged.

Note: This conversion is losseless when it succeeds, but it can fail.
